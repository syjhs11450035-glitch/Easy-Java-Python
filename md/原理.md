# 🧠 EJP 運作原理 (Deep Dive)

本文件將帶領開發者深入了解 EJP IDE 核心引擎的運作邏輯，從代碼轉譯到二進位壓縮技術。

---

## 1. 轉譯引擎邏輯 (The Transpiler)

EJP 並非直接由機器碼執行，而是透過一個高效的 **預處理轉譯器** 將其轉換為 Python 字節碼。

### 1.1 結構化斷句
由於 Python 依賴縮排（Indentation），而 EJP 傾向於 C/Java 的區塊語法（Braces），引擎會進行以下處理：
1. **標記掃描**：偵測所有的 `{`、`}` 與 `;`。
2. **自動換行**：將連在一起的語法（如 `if(cond){log(1);};`）拆解為獨立行。
3. **縮排注入**：根據 `{` 的嵌套深度，自動在轉譯後的 Python 代碼前插入對應數量的空格。

### 1.2 作用域管理
為了確保 `py{}` 塊與 EJP 變數能夠互通，我們使用了一個全域字典 `vars_dict`：
* **EJP 變數**：透過 `sv()` 函數寫入 `vars_dict`。
* **Python 訪問**：在 `py{}` 塊中，開發者可以直接操作 `vars_dict["var_name"]` 來讀取或修改 EJP 狀態。

## 2. 6-bit 二進位壓縮系統

為了減少檔案體積並實現某種程度的原始碼混淆（Obfuscation），EJP 採用了自定義的 **6-bit 編碼協定**。

### 2.1 為何使用 6-bit？
標準字元使用 8-bit (0-255)，但編程常用的字元（a-z, 0-9, 符號）其實不超過 64 個。
* **空間節省**：理論上可比純文本減少約 **25%** 的檔案體積。
* **編碼映射表**：
  * `0-25`：a-z
  * `26-35`：0-9
  * `36-56`：常用符號 (如 `;`, `{`, `%`)
  * `60-62`：關鍵字壓縮（如 `Ⓐ` 代表 `import`）

### 2.2 擴充字元處理 (Unicode)
當遇到不在 64 個常用字元內的符號（如中文、表情符號）時，EJP 會觸發 **Escape 機制**：
1. 寫入一個特殊標誌位元 `63`。
2. 隨後跟隨三個 6-bit 區塊，共同組成一個 18-bit 的 Unicode 數值。

## 3. 畫布渲染原理 (Canvas Implementation)

EJP 的 `canva` 系統採用了 **Tkinter 像素網格技術**：
* **座標映射**：將 EJP 的 `(x, y)` 座標映射到 GUI 的 `Grid` 佈局單元。
* **即時更新**：利用 `root.update()` 在循環中強制刷新界面，實現流暢的像素動畫效果。

---

## 4. 混寫系統 (Polyglot Integration)

EJP v1.0 beta 的終極目標是多語言共存：
* **Python 塊**：透過 `exec()` 動態執行。
* **未來展望**：我們計畫透過 Python 的 `subprocess` 模組，調用系統編譯器來實作 C/C++ 語法塊的即時編譯與鏈結。
